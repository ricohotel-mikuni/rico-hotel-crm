# 大栄商事株式会社 統合管理システム — 開発方針

## 最重要ルール: 実装前に必ず提案書を作成し、承認を得る

新機能・改修を問わず、実装に着手する前に必ず以下を含む提案書を **Artifact** として作成し提示すること。ユーザーが明示的に **「承認します」** と言うまで実装を開始しない。

1. UI/UX改善提案書（Artifact）
2. サイトマップ
3. 画面遷移図
4. ワイヤーフレーム（画面イメージ）
5. 改善理由
6. 実装範囲
7. 実装しない範囲
8. 影響範囲
9. 工数（概算）

小さな修正であっても省略しない。例外はユーザーが明示的に緊急のバグ修正等を指示した場合のみ（その場合も事後に変更内容を報告する）。

## システム設計方針

大栄商事株式会社を頂点とするERPシステムとして設計する。単なるホテル管理システムではない。

```
大栄商事株式会社
└── 事業
    ├── ホテル事業 → 各ホテル（現在: リコホテル三国）
    └── 賃貸事業 → 各物件（未着手）
```

この3階層構成を維持する。中間画面・ホーム画面を不必要に増やさない。

## ナビゲーション規則（全画面共通）

- 左上に「← 1つ前へ戻る」ボタンを設置する。ブラウザの戻るではなく、システム内のロジカルな親画面へ戻る仕様とする。
- パンくずリスト（例: 🏢大栄商事 ＞ 🏨ホテル事業 ＞ リコホテル三国 ＞ 営業管理）を全画面に実装する。各階層はクリックでその画面へ遷移する。
- 戻るボタンの名称・デザイン・サイズ・配置は全システムで統一する。

## 管理センター（管理者専用・一般スタッフには非表示）

銀行・融資管理／書類管理（全社横断）／システム管理／システムアーカイブ／権限管理／監査ログ／バックアップ を集約する。

## 業務フロー・将来拡張（設計時に必ず考慮する）

- **営業**: 営業先 → 案件 → 見積書 → 提案書 → 契約 → 請求書 → 売掛金管理（将来） → 入金管理 → 売上分析。将来的に請求書→入金の自動連携まで拡張できる構成にする。
- **ホテル運営**: フロント／清掃／朝食／夕食／設備／シフト。朝食・夕食は将来 メニュー／在庫／発注／原価／売上 まで拡張できる設計にする。
- **書類管理**: 営業書類（見積書・提案書・契約書・請求書・領収書）とフロント書類（請求書・領収書・宿泊証明書・利用明細）は一覧・管理を分離する。管理者のみ全社横断検索・一覧が可能。
- **帳票共通機能**（全書類に標準で必要）: A4対応・PDF生成・保存・印刷・印刷前PDFプレビュー・メール送信・再発行・発行履歴・インボイス制度対応。
- **将来拡張予定**（追加しやすい構造で設計する）: 在庫管理／電子署名／電子印影／テンプレート管理／メール送信履歴／AI文章作成／AI売上分析／AI営業分析。

## 最終目標

ホテル・賃貸・営業・社員・電子承認・書類・銀行・経営・AIを一元管理する、大栄商事全体のERPシステムとして設計・実装する。
