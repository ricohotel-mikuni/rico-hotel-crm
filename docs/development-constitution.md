## 前文

大栄商事株式会社は、リコホテル三国を端緒とするホテル事業に加え、将来の賃貸事業・営業・人事・電子承認・書類・銀行・経営・AIを一元管理する統合管理システム(以下「本システム」)の長期運用を前提とし、その開発を場当たり的な機能追加の積み重ねではなく、一貫した設計原則に基づく体制へ移行する。

本憲章は、本システムの開発に携わるすべての者(開発を代行するAIエージェントを含む)が従うべき基準を定める。個別の機能提案・実装判断は、常に本憲章に矛盾しないことを確認したうえで行う。本憲章に定めのない事項、あるいは本憲章と個別の指示が矛盾する場合は、大栄商事株式会社代表者の承認をもって本憲章自体を改定することを原則とし、個別の例外運用を積み重ねない。

本憲章は、既存の `CLAUDE.md`(標準開発方針)を置き換えるものではなく、その内容を体系化・明文化し、今後の唯一の基準として拡張するものである。

## 第一章 総則

### 第1条(目的)
本憲章は、大栄商事ERPの長期的な一貫性・拡張性・安全性を確保し、開発の意思決定を属人的判断から本憲章に基づく判断へ移行することを目的とする。

### 第2条(適用範囲)
本憲章は、本システムを構成するすべての画面・機能・データベース・インフラ構成、および今後追加されるすべてのモジュール(ホテル事業・賃貸事業・営業・人事・電子承認・書類管理・銀行融資管理・経営分析・AI機能を含む)に適用する。AI関連機能の詳細原則は「大栄商事AI開発憲章」に別途定め、本憲章と一体をなす。

### 第3条(本憲章の効力)
承認後の本憲章は、本システムに関するすべての開発判断に優先する。個別の依頼・思いつき・過去の一度限りの判断は、本憲章に反する場合これを採用しない。本憲章に反する提案が必要と判断される場合は、機能提案に先立って本憲章の改定を提起する。

### 第4条(改定)
本憲章の改定は、版番号(Ver.)を進め、本文末尾の改定履歴に理由とともに記録する。マイナーな文言修正はVer.1.1のように第2位を、章構成や原則そのものの変更はVer.2.0のように第1位を進める。改定は大栄商事株式会社の承認をもって発効する。

## 第二章 システム設計原則

### 第5条(組織階層モデル)
本システムは「大栄商事株式会社 → 事業(ホテル事業・賃貸事業) → 拠点(リコホテル三国 等)」の3階層構成を維持する。この階層を崩す中間ホーム画面・重複するトップ画面を新設してはならない。

### 第6条(モジュール拡張パターン)
新しい業務機能は `src/modules/<id>/` 配下に実装し、該当レジストリ(`registry.js`)へ1エントリを追加することで有効化する。未着手のモジュールは `status: 'soon'` として先にレジストリへ登録してよいが、実装が伴わないうちは一般利用者に機能があるように見せてはならない(`ComingSoon`表示を用いる)。

### 第7条(画面設計の一貫性)
1画面は1つの業務責務を持つ。既存画面と機能が重複するホーム画面・ダッシュボードを新設してはならない。真に新しい機能を追加する場合も、まず既存のハブ画面(会社ホーム／拠点ホーム／管理センター)への統合を検討し、それで表現できない場合に限り新規モジュールとする。

## 第三章 UI・UX原則

### 第8条(共通ヘッダー・パンくず・戻るボタン)
全画面は単一の共通ヘッダー実装を使用し、独自のヘッダーや2つ目の戻るボタン実装を新設してはならない。左上の「1つ前へ戻る」ボタンはブラウザの戻るではなく、システム内の論理的な親画面へ遷移する仕様を維持する。パンくずリストは各階層をクリックでその画面へ遷移できるものとし、既存のモジュールレジストリから機械的に生成する(パンくず専用の別データを持たない)。

### 第9条(読み込み・エラー表示)
画面の外枠(ヘッダー・パンくず・戻るボタン)は、データ取得の成否に関わらず常に描画する。読み込み中・エラー・空状態は、画面内のデータ領域のみを差し替える形で表現し、画面全体をローディング表示やエラー画面に置き換えてはならない。

### 第10条(アクセシビリティ・モーション)
ロゴアニメーション・演出的なトランジションは、`prefers-reduced-motion`設定を尊重し、無効化した代替表示を必ず用意する。演出は補助的なものにとどめ、演出が完了しなければ操作できない設計を避ける(タップ／クリックでの早送りを許容する)。

### 第11条(配色・トークン)
色指定は `src/lib/constants.js` の `C`(navy / gold 等)を用い、画面ごとに個別の色コードを新規に定義しない。新しい色が必要な場合は `C` 定数へ追加したうえで全画面から参照する。

### 第12条(正式ロゴの使用)
正式ロゴ(Aマーク＋大栄商事株式会社＋DAIEI SHOJI CO., LTD.＋左右の金色ライン。リコホテル三国はアーチ＋Rマーク＋RICO HOTEL＋ー MIKUNI ー)は、文字・英字・金色ラインを絶対に削除・トリミングしない。ログイン画面・会社ホーム・ホテルホーム・サイドバー・ヘッダー・各画面は、必ず正式ロゴ(全要素込み)を使用する。

> 過去にクロップされたAマーク単体版を汎用ロゴとして使い回していたことが問題として指摘された経緯がある。同じ後退を繰り返さないことを本条で改めて明文化する。

### 第14条(比率の保持)
ロゴの縦横比は変更しない。表示枠に合わせて引き伸縮・トリミングせず、`object-fit: contain` 等、原寸比率を保つ表示方法を用いる。

## 第五章 認証・ログイン規定

### 第15条(パスワード認証を基盤とする)
本システムの認証基盤はSupabase Authによるメールアドレス・パスワード認証とする。PIN・生体認証等の簡易認証手段は、この基盤を置き換えるのではなく、信頼済み端末上でのみ追加的に重ねる。

### 第16条(PINの性質)
PINは新しいセッションを発行する認証手段ではなく、信頼済み端末に既に残っている正規セッション(refresh token)を再提示するための追加ゲートとして設計する。将来のWindows Hello・指紋認証・Face ID・Touch ID・二段階認証も同じ位置づけ(既存セッションへの追加ゲート)で設計し、認証基盤そのものを個々の手段ごとに作り替えない。

### 第17条(秘匿情報の扱い)
PIN・パスワード等の秘匿情報は、平文はもちろんハッシュ値もクライアントへ一切返却しない。照合・発行はサーバー側でのみ行う。

> 本システムにはEdge Function・service-role基盤が存在しないため、当面はPostgreSQLの`SECURITY DEFINER`関数(`can_approve()`と同じ確立済みパターン)を照合の唯一の入口とする。Edge Function基盤を新設する場合も、この「サーバー側のみが秘密情報に触れる」原則自体は変更しない。

### 第18条(ロックアウト基準)
簡易認証(PIN等)は5回連続の誤り入力で30秒間ロックする。ロックが2回連続で発生した場合は、簡易認証を継続させず、パスワード認証への差し戻しを必須とする。この数値基準を変更する場合は個別画面の実装ではなく本憲章の改定によって行う。

### 第19条(ログイン履歴の記録義務)
パスワード・PINを問わず、すべての認証の成功・失敗はログイン履歴として記録し、管理者が管理センターから確認できる状態を維持する。記録はクライアントからの直接書き込みを許さず、サーバー側関数を経由させることで自己申告による偽装を防ぐ。

## 第六章 権限・データモデル原則

### 第20条(ロールベースアクセス制御)
権限は「ロール × モジュール × 操作(閲覧・編集・削除・承認・ダウンロード・CSV・印刷)」の行列で管理する新方式(`usePermission`/`PermissionContext`)を標準とする。`user_profiles.role`による単一ロール方式は営業モジュールに残る既存互換のためのものであり、新規機能はすべて新方式を用いる。

### 第21条(複数部署・複数役割の原則対応)
社員が複数の部署・役割を兼務できることを前提にデータモデルを設計する。今後追加するすべての画面・機能は、社員が単一の部署・役割にのみ属するという仮定を置いてはならない。

### 第22条(フェイルクローズ原則)
権限判定に失敗・エラーが生じた場合は、常に「権限なし」側へ倒す(フェイルクローズ)。ネットワーク不調やマイグレーション未適用等を理由に、誤って権限を付与する実装をしてはならない。

### 第23条(リアルタイム反映)
管理者が権限を変更した場合、対象社員は再ログインを要求されることなく反映される状態を維持する(Postgres Realtimeによる購読)。

## 第七章 データベース・マイグレーション規定

### 第24条(冪等性)
すべてのマイグレーションファイルは、何度実行しても・どの状態から実行しても安全であるように書く。

1. `CREATE TABLE IF NOT EXISTS` / `ADD COLUMN IF NOT EXISTS` を用いる
2. `CREATE INDEX`は必ず`IF NOT EXISTS`を付す
3. `CREATE POLICY`の前に必ず同名ポリシーの`DROP POLICY IF EXISTS`を置く
4. `CREATE TRIGGER`は`CREATE OR REPLACE TRIGGER`とする
5. 列を減らす／型を変える`VIEW`再定義の前には`DROP VIEW IF EXISTS`を置く

### 第25条(RLS必須)
新設するすべてのテーブルは`ROW LEVEL SECURITY`を有効化する。閲覧・書き込みポリシーを設けない場合(秘匿情報テーブル等)は、それが意図的であることをコメントで明記する。

### 第26条(特権書き込みはSECURITY DEFINER経由)
クライアントに直接許可すべきでない書き込み(ログイン履歴・監査ログ・PIN照合等)は、テーブルへの直接ポリシーを与えず、`SECURITY DEFINER`関数を唯一の入口とする。Edge Function・service-role基盤が整備されるまでは、この方式を標準の代替手段として用いる。

### 第27条(スキーマキャッシュ通知)
マイグレーションファイルの末尾には`NOTIFY pgrst, 'reload schema';`を付し、PostgRESTのスキーマキャッシュに起因する事故を防ぐ。

## 第八章 開発フロー・承認プロセス

### 第28条(提案書提出の原則)
画面変更・デザイン変更・UI変更・UX変更・画面遷移変更・ロゴ変更・アニメーション追加・カラー変更・新機能追加・ダッシュボード追加に該当する変更は、着手前に必ず提案書を作成し提示する。提案書には次の8項目を含める。

1. 現在の画面
2. 改善案
3. ワイヤーフレーム
4. デザインモックアップ(HTMLプレビュー／Artifact、実際に動作する形を原則とする)
5. 画面遷移図
6. 変更点一覧
7. メリット・デメリット
8. 実装範囲

### 第29条(承認の要件)
提案書に対し、大栄商事株式会社が「承認します」「実装してください」「実装OK」のいずれか、またはこれらに準ずる明確な承認の意思を示すまで、コード変更・データベース変更・画面変更・デザイン変更・実装を一切開始しない。小さな修正であっても本条を省略しない。

### 第30条(緊急時の例外)
大栄商事株式会社が明示的に緊急のバグ修正等を指示した場合に限り、提案書提出前の着手を認める。この場合も、着手後速やかに変更内容を報告する。

### 第31条(本憲章と個別提案の関係)
個別の提案書は本憲章に矛盾しない範囲で作成する。本憲章に反する提案が業務上必要と判断される場合は、当該提案に先立って本憲章の改定(第4条)を提起し、承認を得たうえで提案書を作成する。

## 第九章 拡張性原則

### 第32条(将来業務フローの考慮)
設計時には次の将来拡張を常に考慮する。

1. 営業: 営業先→案件→見積書→提案書→契約→請求書→売掛金管理→入金管理→売上分析(将来的に請求書→入金の自動連携まで拡張可能な構成とする)
2. ホテル運営: フロント／清掃／朝食／夕食／設備／シフト(朝食・夕食は将来メニュー／在庫／発注／原価／売上まで拡張可能な設計とする)
3. 書類管理: 営業書類とフロント書類の一覧・管理を分離し、管理者のみ全社横断検索・一覧を可能とする
4. 帳票共通機能: A4対応・PDF生成・保存・印刷・印刷前プレビュー・メール送信・再発行・発行履歴・インボイス制度対応を全書類の標準機能とする

### 第33条(将来モジュールの設計原則)
在庫管理・電子署名・電子印影・テンプレート管理・メール送信履歴・AI文章作成・AI売上分析・AI営業分析は、追加しやすい構造(第6条のモジュール拡張パターン)で今から設計する。既存モジュールの実装が、これらの将来追加を妨げる一回限りのハードコードにならないよう留意する。
